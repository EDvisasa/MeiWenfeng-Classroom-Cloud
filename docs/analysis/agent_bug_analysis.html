Agent Executor: The Deep Module

在本项目中，AgentExecutor 是一个极深的模块（Deep Module）。它对外暴露了一个简单的接口（接收用户的消息列表，返回处理好的文本流和工具调用状态），而在其内部（Implementation）则包裹了复杂的 LLM 协议、工具解析、XML 标签实时拦截、以及多轮（Multi-turn）思考-行动的闭环。

这使得前端只需接收流式事件（SSE），无需关心工具是如何在后端物理环境中执行的。然而，在此次 </thought> 标签泄漏事件中，后端 Prompt 的约束、状态机缓冲的边界（Seam）以及前端的正则匹配器之间发生了一次精妙的“逻辑错位”，导致了内部实现细节泄漏到了前端 UI。

Problem (泄漏成因)

  • Prompt 冲突： 强制要求 STOP AND WAIT 导致 LLM 截断了结尾的闭合标签。
  • 历史污染： 残缺的 <thought>...<call_tool> 被原样持久化到了多轮上下文中。
  • 模型太聪明： 第二轮时，模型为了语法正确，主动输出了 </thought> 来填补历史坑洞。
  • 前端正则局限： 正则表达式要求成对匹配，无法消化这种跨越了请求边界（Seam）的孤儿标签。

Solution (修复方案)

  • 治本（Backend）： 修改 Agent Prompt，要求 LLM 在输出 <call_tool> 之前必须先闭合 </thought>。
  • 容错（Backend）： AgentExecutor 在持久化 full_content 时，自动检测并补齐未闭合的标签。
  • 兜底（Frontend）： 在 ChatPanel.jsx 进行正则提取前，全局暴力替换清除孤立的 </thought> 标签。

Architectural Takeaway (架构启示)

这个 Bug 完美展示了在 LLM Agent 系统中，状态边界（State Seam）的脆弱性。大模型的无状态特性要求我们必须通过 messages 数组不断回传上下文。当流拦截器（Interceptor）在中途截断并执行副作用（Side Effects）时，如果遗留给大模型的上下文不是语法完备的（Syntactically Complete），大模型的“续写本能”就会在下一个回环（Loop）中产生不可控的自发修正行为。
