问题： LLM 的实现将其预训练的偏好跨越了接缝（seam）发生泄漏，把调用动作嵌套在了思考块（thought）内部，从而破坏了接口（interface）的契约。

解决方案： 深化（Deepen）提示词接口，明确规定在执行调用动作之前的确切闭合顺序。

收益 (Wins)

  • 局部性（Locality）：状态边界得到尊重，错误被控制在单一模块内。
  • 杠杆（Leverage）：用一种统一的提示词模式来约束 N 种本地模型。
  • 接口收缩，实现部分吸收了浅层包装的复杂性。

提示词接口与模型实现的分析

AgentExecutor 为工具调用接缝（seam）定义了极其精确的接口（interface）。提示词明确制定了契约：

TOOL EXECUTION FORMAT:
<thought>
I need to check the current directory contents.
</thought>
<call_tool name="execute_bash">
<command>dir</command>
</call_tool>

该接口要求工具调用必须位于思考块的 外部。提示词从未设计过嵌套结构。

然而，本地 LLM 适配器作为一个有泄漏倾向的实现（implementation），其训练数据中常包含工具动作被嵌套在推理过程内部的偏好。该适配器无视了接口的契约，强行将两者合并在了一起。

此外，接口还强制要求在输出工具调用后必须 立刻停止并等待 (STOP AND WAIT)。LLM 适配器在局部遵循了这一停止指令，导致输出流在生成闭合标签之前就被强行切断。接口失败的根本原因在于，适配器未能在此接缝处满足严格的顺序约束。
