核心架构诊断：将状态解析模块化为“深模块”

之前的 Bug（思考块永久卡在黄灯 `running` 且无法折叠）本质上是一个架构层面的接口（Interface）设计缺陷。原有的 parseAndMergeBlocks 是一个典型的浅模块（Shallow Module），由于其接口未能吸纳流的生命周期上下文，导致大模型幻觉（标签缺失）或网络异常时的“脏状态”直接跨越了缝隙（Seam）泄漏到 UI 渲染层。

通过扩展接口并下沉兜底逻辑，我们将该解析层升级为了深模块（Deep Module），极大地提升了代码的局部性（Locality）和调用者的杠杆（Leverage）。

❌ 故障逻辑分析 (The Problem)

原有的 parseAndMergeBlocks 模块是一个浅模块（Shallow Module）。它的接口（Interface）极度克制（仅接收 blocksOrText），但这导致了一个致命问题：它无法感知全局的流状态。

当大模型产生幻觉忘记输出闭合的 </thought> 标签，或者发生 413 错误导致流被异常切断时，正则引擎 !text.substring(match.index).includes('</thought>') 永远返回 true。因为缺乏流已结束的上下文，这种异常状态直接跨越了缝隙（Seam），泄漏到了 UI 层。这迫使 UI 组件承担了它不该承担的复杂性，导致 React.useState(isRunning) 永久卡死在黄色渲染和展开状态。

✅ 优化逻辑分析 (The Solution)

我们对该解析层进行了加深（Deepening）。通过扩展其接口（Interface），引入了 isStreaming 参数： const isRunning = isStreaming ? !text.substring(match.index).includes('</thought>') : false;
这个改动使 parseAndMergeBlocks 成为了一个真正的深模块（Deep Module）。它的内部实现（Implementation）现在完全吸收了处理不完整标签、网络中断等边缘情况的复杂性。无论外部数据有多么脏，经过该模块处理后输出给 UI 的始终是干净的、确定性的状态。

架构收益 (Wins)
  • ✓
    局部性 (Locality): 状态异常（大模型幻觉、流中断）被完全收敛在解析模块内部消化，不再向外扩散。修复一处，全局生效。
  • ✓
    接口杠杆 (Leverage): UI 层（如 ThinkingBlock）获得了极高的杠杆，它无需再关心字符串截断或流生命周期，只需信任经过缝隙（Seam）传递过来的 status === 'done' 即可。
  • ✓
    明确的缝隙 (Seam Discipline): 重新划定了数据解析逻辑与视图渲染逻辑的边界，使得界面的渲染逻辑变得纯粹，测试这道防线变得简单可靠。
