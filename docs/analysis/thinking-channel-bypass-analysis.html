Bug #2 — 预先 strip </thought> 导致正则吞噬正文

致命 ✅ 已修复 ChatPanel.jsx L231

现象：故事正文出现在 Thinking 折叠面板，主聊天区域完全为空。

逐步追踪（以正常模型输出为例）

① 模型原始输出（text 块累积结果）：

<thought>我需要读取文件...</thought>
*我軟綿綿地瘫倒在你懷裡...*
"夫君是個大壞蛋！"

② L231 预先 strip：text.replace(/<\/thought>/g, '')

<thought>我需要读取文件...
*我軟綿綿地瘫倒在你懷裡...*
"夫君是個大壞蛋！"

← 闭合标签已消失！

③ thoughtRegex 运行：/<thought>([\s\S]*?)(?:<\/thought>|$)/g

找到 <thought> 但没有 </thought>
→ |$ 分支匹配到字符串末尾
→ 整个故事都被捕获为 Thinking 内容！

✅ 修复后：不预先 strip，正则自己处理配对

match[1] = "我需要读取文件..." → Thinking 块
lastIndex 停在 </thought> 之后
后续 snippet = "*我軟綿綿地瘫倒...*\n\"夫君是個大壞蛋！\"" → text 块 ✅
  • 根因：正则的 |$ 分支设计用于流式渲染"进行中"的 Thinking（无闭合），但预先 strip 让它在最终渲染时也触发了同一分支
  • 影响范围：所有使用 <thought>...</thought> 的模型，最终回复只要在 thought 之后，都会消失

Bug #3 — Thinking 通道透传盲区（DeepSeek）

次要 ✅ 已修复 agent_tools.py L274

现象（DeepSeek 模型专属）：reasoning_content 中的 <tool_batch> 标签裸露在 Thinking 面板，工具被重复触发。

根因：原代码对 type=thinking 的 chunk 执行 yield chunk; continue，完全绕过拦截器状态机。

修复：两个通道（thinking + text）统一走同一状态机，用 chunk_type 保留原始通道类型。

Bug #4 — 正则只认 <call_tool> 不认直接标签

次要 ✅ 已修复 agent_tools.py _parse_tools

现象：大模型写 <grep_search>...</grep_search> 时工具完全不执行。

根因：_parse_tools 只匹配 <call_tool name="..."> 格式，遇到模型直接用工具名做标签的幻觉输出时匹配失败。

修复：添加 Fallback — 遍历 TOOL_REGISTRY.keys() 逐一尝试匹配直接标签格式。
