# 学习功能优化路线 (Learning System Roadmap)

> 本文档提取自早期“媚吻锋随身课堂”的核心设定，并深度融合了 Matt Pocock 的 `teach` Skill 架构，旨在为 `MeiWenfeng-Classroom` 未来的“交互式教学功能”提供架构与路线指引。

## 一、核心教学哲学 (The Philosophy)

未来的教学功能不再只是简单的“你问我答”，而是要建立**真正的长期记忆与技能护城河**。我们需要区分两种状态：
1. **流畅度 (Fluency)**：用户当下的顿悟，容易产生“我会了”的错觉。
2. **存储强度 (Storage Strength)**：长期的记忆留存。

为达成“存储强度”，教学模块必须引入**“适度困难 (Desirable Difficulty)”**：
- **提取练习 (Retrieval Practice)**：强制用户回忆。
- **间隔重复 (Spacing)**：配合现有的 ChromaDB 记忆衰减曲线，在不同时段重提旧知识。
- **交叉学习 (Interleaving)**：在考核中穿插不同但相关的技能点。

## 二、工作区与状态管理映射 (Workspace Mapping)

借鉴 `teach` Skill 的结构，未来我们的 Web 工程将把纯文本文件转译为数据库和 UI 模块：

| 原始 `teach` 概念 | 本项目 (MeiWenfeng-Classroom) 的未来实现方案 |
|-------------------|----------------------------------------------|
| `MISSION.md` (学习动机) | **宏大目标管理**：由 `/set_mission` 指令控制，展示在右侧导师状态栏顶端，作为所有课程生成的总纲。 |
| `learning-records/` (学习日志) | **学习决策记录 (LDR)**：利用 SQLite 存储每次突破的关键认知（Aha moment），用于计算用户的**最近发展区 (ZPD)**。 |
| `lessons/*.html` (单节微课) | **交互式 Markdown/React 卡片**：废弃静态 HTML。后端实时生成具有高亮、沙盒运行或测验表单（Checkbox/Input）的 React 组件流。 |
| `reference/` (速查表/知识卡片) | **精炼知识库 (RAG Reference)**：生成的速查表直接沉淀进专用的 ChromaDB Collection，随时供模型或用户调用。 |
| `assets/` (复用资产) | 前端现有的 UI 组件（如 `ChatPanel` 内嵌的小工具）将被复用为测验题板、流程图渲染器等。 |

## 三、阵法思维与反馈循环 (The "Skill" & Feedback Loop)

这是早期媚吻锋设定的精华，也与现代 Agentic 理念不谋而合。
在未来的教学中，**“技能 (Skill) 就是固化的阵法”**。当用户面对一个复杂 Bug 或新语言特性时：

1. **拒绝盲目**：不要让 AI 直接给答案。
2. **建立反馈循环 (Feedback Loop)**：引导用户写出“必定报错的测试脚本”或“最小复现代码”，让系统能通过这个环路不断验证对错。
3. **闭环交互**：AI 的任务是指导用户执行 `Hypothesise -> Instrument -> Test` 的严密循环，这正是我们目前 `AgentExecutor` 拥有本地执行能力（`execute_bash`）后能发挥的最大威力。

## 四、具体实施路径预判 (Next Steps)

1. **改造 `/lesson` 逻辑**：
   将大段文字输出改为“**短内容 + 测验组件**”。AI 返回的不再是万字长文，而是包含 JSON 格式 Quiz 的特殊标签（供前端渲染出选择题或填空题 UI）。
2. **闭环判题机制**：
   用户点击提交后，触发类似 `/submit` 的机制。如果答案不符合要求（或者代码跑不通），触发 AI 的引导性反馈（而不是直接给答案）。
3. **ZPD 追踪模块**：
   在 SQLite 的 `course_progress` 表中加入用户能力评估指标，让系统每次备课时自动判断：现在的难度是太枯燥了，还是太受挫了？
