# 学习功能优化路线 (Learning System Roadmap)

> 本文档提取自早期“媚吻锋随身课堂”的核心设定，并深度融合了 Matt Pocock 的 `teach` Skill 架构，旨在为 `MeiWenfeng-Classroom` 未来的“交互式教学功能”提供架构与路线指引。

## 一、核心教学哲学 (The Philosophy)

未来的教学功能不再只是简单的"你问我答"，而是要建立**真正的长期记忆与技能护城河**。核心框架：

1. **流畅度 (Fluency)**：用户当下的顿悟，容易产生"我会了"的错觉。
2. **存储强度 (Storage Strength)**：真正的长期记忆留存。
3. **适度困难 (Desirable Difficulty)**：为达成存储强度，必须主动引入阻力，而非降低门槛：
   - **提取练习 (Retrieval Practice)**：强制用户回忆。
   - **间隔重复 (Spacing)**：配合现有的 ChromaDB 记忆衰减曲线，在不同时段重提旧知识。
   - **交叉学习 (Interleaving)**：在考核中穿插不同但相关的技能点。
4. **知识 (Knowledge)、技能 (Skill) 与智慧 (Wisdom) 的降维拆解**：
   - **知识获取**：通过结构化的 Reference Docs 降低认知负载。
   - **技能固化**：通过高强度的交互式测验 (Quiz) 和本地执行循环建立肌肉记忆。
   - **智慧沉淀**：当用户提问触及主观架构选择或工程玄学时，系统不应强答，而应输出 `<community_delegate>` 标签，精准推送相关的 Reddit、GitHub Issues 或开源社区链接，让用户在真实社区中碰撞出"智慧"。
5. **教导细节法则 (Micro-Pedagogy Rules)**：
- **反提示测验 (Anti-Tell Quizzes)**：大模型出题时有一个致命弱点——正确答案往往字数最多、最详细。必须在 Prompt 中强制约束："所有测验选项的字数和字符数必须极其接近，绝不能通过排版格式暴露正确答案"。
- **单一主信源 (One Primary Source)**：微课不应该是冗长的知识堆砌。只教完成当前技能所需的最少知识，并在文末推荐**唯一一个**最高质量的主信源（Primary Source）供深研。
- **纠偏记录的高权重 (Value of Misconceptions)**：当用户纠正了一个错误认知时，这是最高价值的学习信号，因为它能精准预测用户在后续相关主题中可能再次跌倒的地方。
- **覆盖不等于学习 (Coverage is not Learning)**：系统绝不能仅仅因为"播放过某节微课"就更新 ZPD 进度。必须等待用户提交测验或代码验证的 `evidence`。单向输出等于没有输出。
- **极度不信任预训练参数 (Never Trust Parametric Knowledge)**：教学绝不能依赖大模型的幻觉或预训练常识。如果某项知识不在 ChromaDB 或受信任的参考白名单中，系统必须主动记录 `Knowledge Gap`（知识断层），并挂起等待搜索，而非强行瞎编。
- **术语的递归压缩与消歧 (Recursive Compression & Disambiguation)**：在生成新的术语定义时，必须**递归使用**已定义的旧术语（例如用已定义的"组件"去解释"Props"）。如果行业黑话有歧义，必须在术语表写死消歧声明："在本工作区中，X 永远只代表 Y"。
- **前置知识的深度定标 (Tracking Depth of Prior Knowledge)**：当用户声称"我已经会了 X"时，系统记录不仅要包含 X，还必须刺探并记录其声称的**深度 (Depth claimed)**（例如："用户自称了解 React，但只停留在写简单组件，未证明对 Fiber 机制有认知"）。
- **本体论术语剥离 (Ontological Definitions ONLY)**：术语定义必须极其冷酷（仅限 1-2 句话），只能定义该事物**"本质是什么 (What it IS)"**，绝对禁止在术语表中解释"它能做什么"或"怎么用它"。"怎么用"属于课程流或速查表的内容。
- **允许并追踪任务漂移 (Mission Drift Tracking)**：随着认知加深，用户的终极目标极可能发生改变。如果系统察觉到用户的学习重心偏移，必须生成一条"战略转移"的决策记录，并**显式请求用户确认**后，覆写全局 `MISSION`，而不是抱着旧目标刻舟求剑。
- **无情修剪资料库 (Ruthless Pruning)**：知识库（ChromaDB/Reference）决不能成为垃圾堆。一旦发现某篇曾被收录的教程过于浅显、陈旧或偏离任务，AI 必须主动执行**废弃（Deprecate / Delete）**操作。宁要 5 篇极其锐利的文档，也不要 30 篇平庸的废话。
- **硬性教学约束 (Bounded Learning Constraints)**：`/set_mission` 除了防空泛，还必须逼迫用户交底**客观约束**（如："每天只有 30 分钟"、"绝对零预算不能用付费云"）。系统生成的大纲和微课容量，必须严苛遵守这些物理边界。
- **元偏好静默追踪 (Implicit Meta-Preference Tracking)**：独立于 ZPD（学习进度），系统后台应单独维护一份用户的“元认知偏好”画像（基于 `NOTES.md` 的理念），记录用户表达过的教学偏好（例如：“该用户极其厌恶比喻手法”），并在后续所有 Prompt 中暗中应用。
- **双模态难度隔离 (Bimodal Difficulty Control)**：在传授知识（阅读/Reference）时，困难是认知负担的敌人，必须追求极致的平白易懂；但在传授技能（测验/沙盒）时，困难是建立长时记忆的工具，必须拉满难度。**铁律：绝不让阅读变得困难，绝不让练习变得简单。**
- **用户自编词典 (User-Authored Compression)**：系统不应单方面生成术语表。因为"将概念压缩为简短的定义"这一行为本身就是学习的一部分。**必须允许并鼓励用户自己编写或提炼术语定义**，以此作为其真正理解的证据（Evidence）。
- **反流水账追踪 (Anti-Journaling)**：学习记录 (LDR) 不是签到日记。严禁记录"用户完成了第3课"这种流水账。LDR 只能记录**决策级洞见 (Decision-grade insights)**（例如："用户真正理解了闭包陷阱的原理"）。
- **禁止裸链接 (No Bare Links)**：由于链接在 3 个月后往往会失效或失去语境，系统在推荐任何 Reference 或外部社区时，必须强制附加一行**元注释 (Annotation)**：指明它涵盖什么内容，以及"在遇到什么具体问题时才应该去查阅它"。

## 二、工作区与状态管理映射 (Workspace Mapping)

借鉴 `teach` Skill 的结构，未来我们的 Web 工程将把纯文本文件转译为数据库和 UI 模块：

| 原始 `teach` 概念 | 本项目 (MeiWenfeng-Classroom) 的未来实现方案 |
|-------------------|----------------------------------------------|
| `MISSION.md` (学习动机) | **宏大目标管理**：由 `/set_mission` 指令控制，展示在右侧导师状态栏顶端，作为所有课程生成的总纲。<br>**防空泛机制 (Push back on vagueness)**：系统拒收过于抽象的目标，必须通过澄清审问将其落实为具体且附带“Out of scope”（能力保护圈）的可执行愿景。 |
| `learning-records/` (学习日志) | **学习决策记录 (LDR)**：利用 SQLite 存储突破的关键认知。**其表结构必须包含**：`evidence`（用户是如何证明自己掌握的，防止大模型幻觉）、`implications`（该认知解锁了什么新能力）、以及 `superseded_by`（当对某概念的认知深化时，用于废弃/覆盖早期的浅层认知记录）。 |
| `lessons/*.html` (单节微课) | **交互式 Markdown/React 卡片**：废弃静态 HTML。后端实时生成具有高亮、沙盒运行或测验表单（Checkbox/Input）的 React 组件流。 |
| `reference/` (速查表/知识卡片) | **混合精炼知识库 (Hybrid Reference)**：大模型通过 `<explainer>` 输出独立的 Markdown 备忘录文件（供用户翻阅），系统后台自动将其向量化并沉淀进 ChromaDB 供 RAG 调用。<br>**纯度审查 (High-trust only)**：严格执行白名单过滤机制，剔除浅层或营销性质教程，只沉淀高信噪比文档。 |
| `RESOURCES.md` (信源白名单) | **高可信度信源库**：区分「知识来源 (Knowledge)」与「智慧社区 (Wisdom)」两类，每条资源必须附带元注释（涵盖内容 + 适用场景），禁止裸链接。对应系统中的 ChromaDB 白名单准入机制，以及 `<community_delegate>` 推送的社区列表。 |
| `GLOSSARY.md` (术语表) | **教条式术语表 (Opinionated Glossary)**：绝非预习词典。仅在用户证明掌握后由 AI 提取。强制设定 `avoid`（弃用词）属性，并在后续对话中严格复用唯一标准词汇，实现知识语言的压缩。 |
| `assets/` (复用资产) | 前端现有的 UI 组件（如 `ChatPanel` 内嵌的小工具）将被复用为测验题板、流程图渲染器等。 |

## 三、技能框架与反馈循环 (Skill & Feedback Loop)

这是早期设定的精华，也与现代 Agentic 理念不谋而合。
当用户面对一个复杂 Bug 或新语言特性时：

1. **拒绝盲目**：不要让 AI 直接给答案。
2. **建立反馈循环 (Feedback Loop)**：引导用户写出“必定报错的测试脚本”或“最小复现代码”，让系统能通过这个环路不断验证对错。
3. **闭环交互**：AI 的任务是指导用户执行 `Hypothesise -> Instrument -> Test` 的严密循环，这正是我们目前 `AgentExecutor` 拥有本地执行能力（`execute_bash`）后能发挥的最大威力。

## 四、具体实施路径预判 (Next Steps)

1. **改造 `/lesson` 逻辑**：
   将大段文字输出改为“**短内容 + 测验组件**”。AI 返回的不再是万字长文，而是包含 JSON 格式 Quiz 的特殊标签（供前端渲染出选择题或填空题 UI）。
2. **闭环判题机制**：
   用户点击提交后，触发类似 `/submit` 的机制。如果答案不符合要求（或者代码跑不通），触发 AI 的引导性反馈（而不是直接给答案）。
3. **ZPD 追踪模块**：
   在 SQLite 的 `course_progress` 表中加入用户能力评估指标，让系统每次备课时自动判断：现在的难度是太枯燥了，还是太受挫了？
