1. 状态与上下文收束 (State & Context Aggregation)

Strong in-process
backend/routers/chat.py
                flowchart TD
                  A[ChatRequest] --> B[CharacterStateManager]
                  A --> C[RAG Retrieval\nMemory & Docs]
                  A --> D[IDE Context]
                  B -.seam.-> E[System Prompt Builder]
                  C -.seam.-> E
                  D -.seam.-> E
                  E --> F[LLM Stream]

Problem: 散落的记忆、RAG 向量检索与动态好感状态需要在送入 LLM 前被集中处理。

Solution: _build_full_system_prompt 作为深模块 (deep module)，吸收了数据库查询、向量库请求与本地文件读取，对外暴露单一系统提示词接口。

  • locality: 数据准备的缺陷集中在此模块
  • interface shrinks; implementation absorbs the wrappers

2. 核心人设 XML 注入 (Core Persona Injection)

Strong local-substitutable
backend/services/prompts.py
                flowchart LR
                  A[load_local_persona] -->|Reads 60KB TXT| B[SYSTEM_PROMPT_TEMPLATE]
                  B --> C{XML Wrapper}
                  C -->|system_directive| D[Final Prompt]
                  C -->|character_persona| D
                  C -->|relationship_context| D

Problem: 直接将 60KB 的媚吻锋原版人设文本暴露给 LLM，会导致指令遗忘或角色崩塌。

Solution: 构建模板化的 XML 缝隙 (seam)。通过 <system_directive> 锁定规则（内心戏格式、非语言动作 * * 渲染），再将纯粹的背景文本塞入 <character_persona> 节点中。

  • leverage: 一个模板支持切换「精简」或「原版」
  • locality: 角色的剧情设定与对话格式被解耦分离

3. 动态属性钩子 (Dynamic State Mutation Hooks)

Worth exploring in-process
backend/services/prompts.py
Affection Conditionals
Social Status / Skills
Refractory Directives
<property_update> Tag Rules

Problem: 角色需随好感等四维属性表现出截然不同的性格，并实时回调后端进行数值更新。

Solution: 在 Prompt 尾部组装动态规则带。基于阈值（如 affection >= 75）硬编码具体行为指令，并设计隐式的 <property_update> 输出格式赋予 LLM 数值修改权。

ADR: 此实现本质上是将 LLM 作为状态机引擎（State Machine），通过逆向控制反转实现后端数据变异。

  • interface: XML 钩子定义了模型向系统的数据输出契约
  • 实现深度: 复杂的行为规则被折叠，无需外部引擎驱动

核心痛点与优化建议 (Recommendation)

目前 get_system_prompt 内充斥着大量的字符串拼接。建议深化该模块 (deepen the module)：将其重构为独立的 Prompt Pipeline 或使用 Jinja2 等模板引擎。这能将业务判定逻辑与文本组装彻底解耦，提高代码测试性 (locality) 并减少重度包装器 (shallow wrappers) 的蔓延。
