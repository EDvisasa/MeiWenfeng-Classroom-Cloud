架构审查：Mission 模块

日期: 2026-06-24

模块 (Module)
接缝 (Seam)
泄漏 (Leakage)

深化 Mission 状态

强烈推荐 进程内
backend/database.py
backend/services/slash_handler.py

痛点 (Problem)

“User Mission (用户目标)” 的概念属于浅层模块 (Shallow) —— 其内部状态泄漏到了三个独立的 SQL 数据表中 (user_mission, mission_draft, learning_decision_records)。

解决方案 (Solution)

将这些数据表折叠成一个单一的深度 Mission 状态机，将草稿状态和漂移历史隐藏在一个高内聚的接口之后。

重构前 (浅层与泄漏)

                flowchart TD
                  A[slash_handler.py] --> B[(mission_draft)]
                  A --> C[(user_mission)]
                  A -.leak.-> D[(learning_decision_records)]
                  
                  classDef leak stroke:#dc2626,stroke-width:2px;

重构后 (深化)

                flowchart TD

                  classDef deep fill:#1e293b,stroke:#0f172a,color:#fff;

收益 (Wins)

  • 局部性 (Locality)：状态变更逻辑集中在一个模块中。
  • 杠杆率 (Leverage)：统一接口，隐藏底层表结构的复杂性。
  • 接口收缩：实现层吸收了冗余的草稿包装器。

核心推荐

深化 Mission 状态

当前的实现迫使 handler 直接编排多个数据表。将其深化为一个统一的接口将大幅提升局部性 (Locality)，并在未来添加新的目标机制时减少摩擦。
