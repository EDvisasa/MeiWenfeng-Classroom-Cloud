深模块解耦与响应管线拦截架构决策

合并收纳原 4 份通用架构改良与验证报告。践行 Matt Pocock “深模块 (Deep Modules)” 与 “无边际逻辑拦截” 哲学，重构数据库状态访问与响应后处理管线。

🛡️ DB-Transaction (核心数据库事务闭环) 🏷️ ZPD-Adaptive (最近发展区难度自适应)
1

演进背景与核心痛点 (Context & Challenges)

浅层模块耦合与接口响应死锁的技术债务

⚠️ 路由层直接操作数据库

旧版代码在 API 路由端点中直接裸写 raw SQL 来读取和修改角色的好感度、亲密度与学习进度。多并发请求下极易引发 SQLite 数据库锁死（Database Locked）和事务未提交异常。

❌ 副作用逻辑阻塞主响应流

在指令响应函数中混杂了长达 500 行的副作用处理逻辑（发勋章、触发讲义、更新 ZPD 曲线）。导致用户发送对话后需等待所有副作用执行完毕才能收到回复，接口超时严重。

2

决策与解决方案 (Decision & Solution)

践行深模块 (Deep Modules) 与异步后处理管线解耦

2.1 设立 CharacterStateManager 深模块

在 character_state.py 中建立统一的状态管理器，封装内部线程锁 _state_lock 与所有 SQLite 事务读写。外部路由层仅通过极简接口调用，封口事务异常。

2.2 ResponsePipeline 异步后处理管线

将庞杂的副作用剥离至独立的 ResponsePipeline 与 side_effects.py 注册表（如 SystemPassHandler, GlossaryHandler），实现主对话毫秒级返回，副作用异步消化。
