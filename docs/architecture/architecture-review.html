代码库架构审查报告

1. 聊天响应管道 (The Chat Response Pipeline)

强烈推荐

backend/routers/chat.py, backend/services/slash_handler.py

存在的问题 (Problem)

解析 AI 输出标签（如 <explainer>, <glossary>）并触发副作用的逻辑，被困在 slash_handler.py 中的 _stream_normal_chat_with_injection 的 finally 块里。如果用户发送不带斜杠命令的普通消息，chat.py 会使用自己的 event_generator，完全忽略这些标签。这导致了严重的局部性缺失和标签丢失风险。

解决方案 (Solution)

创建一个深度的 ResponsePipeline 模块。让 chat.py 和 slash_handler.py 都将它们的 stream_chat 生成器传递给这个管道。该管道负责缓冲流、向前端发送 SSE（服务器发送事件），并在流结束时统一执行副作用。

优势 (Benefits)

  • 局部性 (Locality): 所有的后处理和标签拦截都集中在一个地方。
  • 杠杆效应 (Leverage): 调用者只需要说“流式传输这个聊天并处理副作用”即可。

2. RAG 工厂的透传问题 (The RAG Factory Pass-throughs)

值得探索

backend/services/rag_factory.py, backend/services/*_client.py

存在的问题 (Problem)

rag_factory.py 既充当工厂，又充当透传代理。其中的一些方法仅仅是为了调用 chroma_rag_client 上的方法而复制了读取配置的逻辑，这破坏了工厂模式。这种接口是泄漏的，迫使工厂必须知道哪些后端支持内存存储。

解决方案 (Solution)

定义一个深度的 RAGClient 接口（协议），包含通用方法。让所有的适配器都实现这个接口。rag_factory.py 应该被精简为一个单一的纯工厂函数：get_rag_client()。

优势 (Benefits)

  • 杠杆效应 (删除测试): 删除工厂中的透传方法，可以很好地将复杂性集中到适配器中。调用者获得了一个干净的接缝。
  • 可测试性: 在测试中很容易替换为 MockRAGClient。

3. 代理工具循环 (The Agent Tool Loop)

探索性建议

backend/services/model_router.py, backend/services/agent_tools.py

存在的问题 (Problem)

model_router.py 中的 stream_chat 函数将 HTTP 流式传输、OpenAI 协议路由、XML 工具解析以及工具执行纠缠在一起。这使得在不进行实际 LLM 调用的情况下测试工具解析逻辑变得非常困难。

解决方案 (Solution)

通过提取一个 AgentExecutor 模块来深化 agent_tools.py。stream_chat 应该只负责与 LLM 通信并生成数据块。AgentExecutor 包装 LLM 客户端、拦截流、解析 XML、执行工具，并将结果反馈给 LLM。

优势 (Benefits)

  • 局部性 (Locality): 对工具 XML 解析逻辑的更改不会有破坏核心 LLM 流协议的风险。
  • 可测试性: 可以使用模拟的 LLM 来测试 AgentExecutor。

首推建议 (Top Recommendation)

我建议首先着手处理 1. 聊天响应管道 (The Chat Response Pipeline)。目前分散的聊天处理逻辑是一个活跃的 bug 来源——如果 AI 决定在正常对话中（没有使用 /submit 或 /lesson 命令）使用 <glossary> 标签，系统将无法解析它，该术语就会丢失。整合流处理逻辑将立即稳定应用程序的核心功能。
