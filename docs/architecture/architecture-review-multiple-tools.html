核心架构解析：非思考块作为屏障下的独立区块生命周期

在交互面板（ChatPanel.jsx）的底层数据处理链中，消息解析与合并由 frontend/src/utils/blockParser.js 中的 parseAndMergeBlocks 负责。当 AI 模型的多次工具调用（Tool Calls）穿插在多个思考（Thinking）段落之间时，工具调用块或普通文本块在合并步骤（Merge Step）中天然充当了隔离屏障（Isolation Barrier）。合并逻辑仅对相邻的 thinking 块执行合并操作；一旦中间隔有工具调用，合并链条会被主动截断。因此，每个被屏障隔开的 ThinkingBlock 都会依据自身是否包含闭合的 <thought> 标签来独立计算状态（running 或 done），从而在 UI 上呈现出精准、符合真实执行顺序的多段链式思考与工具调用的交替历史。

🔍 归一化与区块构建 (Step 1: Normalization)

'running' : 'done' }); 这意味着任何一个在流式传输中尚未闭合 `` 标签的思考段落会被精确标记为 `running`，而已经完成输出并闭合标签的段落则被标记为 `done`。

🛡️ 合并步骤的屏障保护机制 (Step 2: Merging)

在第二步合并逻辑（第 170-185 行）中，代码遍历归一化后的数组。当且仅当当前块 `b.type === 'thinking'` 且合并数组的最后一个块 `last.type === 'thinking'` 时，才会触发文本拼接和状态继承（last.status = b.status）。

当 AI 模型执行 思考段落 A -> 执行工具 Tool -> 思考段落 B 时，由于 Tool 块被推入 `merged` 数组，处理思考段落 B 时 last.type 为 'tool'，从而直接进入 else 分支独立添加。非思考块在合并逻辑中起到了完美的物理屏障作用，防止了跨工具调用的思考内容混淆或状态错误污染。

澄清与架构评估 (Architecture Assessment)

历史文档误区更正

  • 误区陈述：旧版本分析曾认为屏障隔离导致“后一段无法继承前一段的状态，使得 UI 指示灯提前变绿”。
  • 事实真相：当流式到达工具调用后的新思考块时，若其自身正在流式生成，其局部状态即为 `running`，UI 会正确渲染动画，不存在提前变绿的问题；若其已生成完毕，显示为 `done` 正是预期行为。

现行架构设计优势

  • 状态隔离清晰：避免了不同阶段思考块的状态相互污染。
  • 历史交互还原：多轮 Tool Calling 流程中的每一次思考和动作都被清晰记录并依序呈现。

核心架构原则 (Core Architectural Principles)

  • 1

    生命周期独立性 (Lifecycle Independence)

    每个非相邻的 ThinkingBlock 的生命周期应当完全由其自身的标签闭合状态决定，确保跨段落间的数据流解耦。

  • 2

    屏障一致性 (Barrier Consistency)

    工具调用（Tool Blocks）作为外部执行的界限，理应成为合并策略中的物理屏障，保障会话轨迹的严格时序。

  • 3

    状态继承的局部性 (Status Inheritance Locality)

    第 178 行的 `last.status = b.status` 仅限在同一连续思考片段被拆分为多个子文本块时聚合使用，防止跨级冒泡。
