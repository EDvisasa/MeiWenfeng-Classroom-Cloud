核心架构诊断：统一生命周期与状态合并逻辑

在当前的 ChatPanel.jsx 中，parseAndMergeBlocks 被升级为了一个依赖 isStreaming 标志的深模块。然而，针对多工具连续调用后紧接普通的 Thinking 过程这一场景，它仍存在设计上的脆弱性。当后端不断发来带状态的 blocks 时，前端将它们合并处理，但对于最后一次大模型补充的 Thinking 过程，原有的合并逻辑可能因未能正确处理前一个已完结状态（done）和新流入状态（running）的层级关系，导致本该处于 running 的思考块在合并后提前被判定为 done（指示灯变绿）。这本质上是状态在合并（Merge）缝隙处的泄露。

深度重构：状态合并缝隙的加深 (Deepening the Block Merge Seam)

Strong Recommendation in-process dependency

❌ 问题分析 (The Problem)

在 parseAndMergeBlocks 的第二步（合并相邻的 thinking 块）中，代码是这样处理状态的： if (b.status === 'running' && isStreaming) last.status = 'running'; 当发生多次工具调用后，模型可能会再次输出 <thought>。如果后端传来的 msg.blocks 结构未能良好地隔离各个思考阶段（例如工具调用前后的 thinking 被压平），或者在处理非文本块（如 tool 块）时： const bStatus = isStreaming ? b.status : (b.type === 'thinking' || b.type === 'tool' ? 'done' : b.status); 如果工具块（tool）插在两个思考块（thinking）之间，它们无法被合并。但如果后端的机制是将后续的文本流追加在之前的文本块中，那么最后一个新产生的 thinking 块的 status 将取决于正则的解析。由于 isStreaming 是全局级别的，如果前一个思考块已经被判定为 done（因为它有 </thought>），而新的思考块正在生成（没有闭合），它们合并时 last.status = 'running' 能够生效。

真正的泄露点在于：如果后端将工具调用过程作为一个独立的 block 发送，而大模型在工具调用完成后的“继续回复”产生的 <thought> 是包含在新的文本流中，但前端将之前已完成的整个消息树视作 done 状态，或者在正则处理中，由于之前的 </thought> 存在，导致 lastIndex 处理出现覆盖，使得新的 running 状态未被正确捕获。

✅ 解决方案 (The Solution)

将 parseAndMergeBlocks 对区块的规整化（Normalization）和合并（Merging）进行更彻底的隔离。

  1. 引入区块级别的生命周期：不仅依赖全局的 isStreaming，还要根据每个思考块是否真正闭合来决定其独有的状态。如果文本块中存在未闭合的 <thought>，且全局正在 isStreaming，那么只有最后那个未闭合的块才是 running。
  2. 强化合并逻辑的杠杆（Leverage）：在合并多个 thinking 块时，状态不应简单地通过 if (b.status === 'running') last.status = 'running' 来覆盖，而应明确：只要合并组中存在任何一个是 running，整个合并后的块就应该是 running；但更优的架构是不合并跨越了工具调用的思考块，让工具调用前后的思考过程独立显示，从而保留时间线上的局部性（Locality）。
