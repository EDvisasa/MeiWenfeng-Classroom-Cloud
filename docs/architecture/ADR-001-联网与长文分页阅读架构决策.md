# ADR-001: 智能体联网检索与长文分页阅读引擎架构决策

- **状态**: 已采纳 (Adopted)
- **时间**: 2026-06-27
- **决策人**: 媚吻锋架构审查团队 / Antigravity

---

## 1. 背景与上下文 (Context)

在媚吻锋智能体（Agent）向外扩展知识边界的过程中，面临两个严峻的底层挑战：
1. **网络内容抓取冗余与噪声大**：直接使用 HTTP 请求拉取的原始网页多为数万行的 HTML 源码，充斥着大量的脚本、广告、导航栏与样式表（CSS），直接塞入上下文会导致大模型显存瞬间爆炸，或触发 LLM API 的 `context_length_exceeded` / `unexpected EOF` 中断。
2. **一次性长文吞吐导致“编造空间（Hallucination）”**：若强行让 LLM 总结超长文本，大模型往往存在“注意力稀疏”现象，容易遗漏网页深处的核心真实数据，并在信息盲区自动编造内容。

---

## 2. 决策方案 (Decision)

我们放弃了引入笨重的无头浏览器（如 Playwright/Puppeteer）与复杂的后台爬虫挂机队列，采用**“轻量检索 + 降维洗净 + 滚动翻页”**的三位一体架构：

### 2.1 纯粹检索层 (`WebSearchTool`)
- **选型**：使用 `ddgs` (DuckDuckGo Search Python Client) 提供免费、开箱即用、免 API Key 的泛搜能力。
- **约束**：强制限制单次返回条数（`max_results=3` 或 5），仅提取标题、超链接与短摘要。

### 2.2 降维正文抓取层 (`ReadUrlContentTool`)
- **选型**：`httpx` + `beautifulsoup4` + `markdownify`
- **处理流**：
  1. `httpx.get` 发起异步/同步 HTTP 请求获取 HTML。
  2. `BeautifulSoup` 强势剔除 `script`, `style`, `nav`, `footer`, `header`, `aside` 等无效噪声节点。
  3. `markdownify` 将洗净后的 DOM 树彻底**降维转化为极高逻辑浓度的纯正 Markdown 文本**。

### 2.3 800行硬截断滚动翻页机制 (Pagination)
- **机制**：复刻 `ReadFileTool` 的设计，对转换后的 Markdown 长文实施单次最高 **800 行**的强制物理截断。
- **心理暗示驱动**：在截断行尾部动态注入带有强烈心理导向的系统提示：
  `[Warning: Webpage content truncated at 800 lines. Total length is {total_lines} lines. 👉 To read more, call this tool again with <start_line>{s + 800}</start_line>]`
  利用大模型自回归生成时渴望补全上下文的本能，激活智能体**主动滚屏阅读长文直至获取全量真相**的心流闭环。

### 2.4 前端沉浸感隔离与映射
- **机制**：在 `AgentExecutor` 层完全静默拦截繁重的文本回显。
- **UI 映射**：在 `ChatPanel.jsx` 的 `ToolBlock` 组件中硬编码展示映射规则：
  - `web_search` -> **`Web`**
  - `read_url_content` -> **`Browser`**
  避免向最终用户暴露底层丑陋的函数命名，捍卫酒馆式 Roleplay 沉浸体验。

---

## 3. 后果与评估 (Consequences)

### 正面影响 (Pros)
- **极速轻量**：无浏览器进程开销，毫秒级抓取与转换。
- **杜绝长文幻觉**：智能体通过 800 行翻页逐步消化长文，真正做到了“读全真相再下定论”。
- **上下文极度省流**：Markdown 正文体积比原始 HTML 缩小 80% 以上。

### 负面与局限 (Cons)
- **动态 SPA 网页限制**：对于高度依赖客户端 JavaScript 渲染的强动态单页应用（如 Twitter/知乎直连），纯 HTTP 抓取可能只能拿到空白壳子或登录拦截提示。若未来高频遇到此类场景，需考虑升级至轻量级 Cloudflare Browser Rendering API。
