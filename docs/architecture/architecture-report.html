Functional Modules Deep Dive

1. AgentExecutor & ResponsePipeline

AgentExecutor: Wraps the raw LLM stream. It injects a strict <tool_use_guidelines> system prompt. It intercepts <tool_batch> tags using its own buffer, pauses the stream to execute tools concurrently, and feeds the [System Tool Result] back to the LLM for the next iteration.

ResponsePipeline: Sits between the AgentExecutor and the Frontend. Its TagStreamInterceptor processes chunks character-by-character. If the LLM generates a side-effect (e.g., <system_pass> to advance a course), it buffers the tag, prevents it from leaking to the UI, and dispatches it to the ActionRegistry at the end of the stream.

2. Memory & Decay System

Avoids infinite context growth. Raw messages are stored in SQLite as level 0. The MemoryDecayHandler (dispatched via ActionRegistry) periodically condenses level 0 logs into level 1 (summaries), and older level 1s into level 2 (milestones). ChromaDB provides semantic similarity search, while SQLite provides recent chronological summaries. Both are injected into the prompt.

3. Model Routing & Gateway

The OpenAILLMClient acts as a LiteLLM-like gateway. Because all interactions are standardized to the OpenAI protocol, switching from an online model (DeepSeek) to a local model (Ollama) requires zero code changes—only a Database update of the base_url and model_id.

4. Course & UI Management

The syllabus and character states (affection, social skills) are stored in SQLite. State changes are triggered by the AI emitting specific XML tags (intercepted by the pipeline). The Frontend UI uses a "Deep Module" React architecture to sync the typewriter state, tool execution loaders, and the dynamic Token Ring indicator.

Example Scenarios

Scenario A: Tool Execution Flow (Bash/Grep)

1. User asks: "Check my config file."
2. AgentExecutor sends prompt to LLM. LLM outputs <thought>...</thought><tool_batch><call_tool name="read_file">...
3. The Executor's buffer catches <tool_batch>, stopping it from reaching the pipeline.
4. Executor parses the AST, runs read_file via python's open().
5. Executor yields tool_start and tool_output dicts. ResponsePipeline formats these as SSE JSON. Frontend renders a spinning loader.
6. Executor appends the result to the message history and triggers iteration 2.
7. LLM reads the result and outputs final text.

Scenario B: Slash Command (/lesson)

1. User types /lesson.
2. chat.py intercepts it before hitting the LLM.
3. handle_slash_command() fetches the current syllabus from the database.
4. It overrides the user message with an invisible system instruction: "The user has started the lesson. Output the course material..."
5. The LLM responds naturally, transitioning the UI into lesson mode.

Scenario C: Memory Retrieval & Decay

1. User says something. ChatRouter queries ChromaDB using the user's text.
2. It also queries SQLite for the last 3 days of Level 1/2 summaries.
3. Both are injected into the <system> prompt.
4. After response, both messages are saved to SQLite (Level 0).
5. If Level 0 count hits 10, the pipeline invokes the MemoryDecayHandler via the ActionRegistry, summarizes the 10 logs using the LLM, saves the summary as Level 1, and deletes the Level 0 logs.

Scenario D: Model Switching

1. User clicks "API Settings" in UI and selects Ollama.
2. Backend updates model_config table: is_active=1 for Ollama, base_url=http://localhost:11434/v1.
3. Next chat request, get_active_model() fetches Ollama's URL.
4. OpenAILLMClient seamlessly routes the request to Ollama using the identical OpenAI API format.
