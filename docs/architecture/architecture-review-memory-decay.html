架构分析报告 — 记忆系统衰减与时间压缩逻辑深度核查

Date: 2026-06-17 | Target: backend/services/memory_decay.py

核心诊断：逐日衰减导致的“单点压缩”假象

经过深度仿真测试与数据库核查，用户的怀疑完全正确。原有的记忆压缩逻辑（Level 1 -> 2 -> 3）存在严重的时间轴滑动窗口 Bug。

原逻辑使用 timestamp < datetime('now', threshold) 进行查询，由于定时任务每天运行，导致每天只有恰好跨越阈值的那 1 条日志被选中。结果是：大模型被要求“压缩这几天的日记”，但实际上每次只被喂给了 1 天的日记。这不仅失去了记忆归纳、提取羁绊的意义，还会导致 Level 2 和 Level 3 的日志数量无限膨胀（1 换 1），且浪费大量 Token。

✅ 修复方案 (The Fix)

我们重写了 backend/services/memory_decay.py 中的 Phase B 逻辑：

  1. 确立绝对锚点：每次执行时，查询该层级下最老的一条活跃日志的 timestamp 作为 oldest_time。
  2. 计算时间窗口：将配置中的阈值（如 "-2 days"）解析为绝对天数，计算出 window_end = oldest_time + threshold_days。
  3. 整体打包：只有当当前时间超越了 window_end 时，才将这期间所有的日志一次性取出，喂给大模型进行合并压缩。这确保了如“近一周”、“近一个月”的 Prompt 能真正发挥出归纳总结的威力。

✅ 附带修复 (Schema Fix)

在审查过程中，同时发现 backend/database.py 中 init_db() 创建 memory_logs 表的 SQL 语句遗漏了 status TEXT DEFAULT 'active' 字段。这会导致如果未来重建数据库，衰减逻辑将直接崩溃。已在本次核查中一并修复。
