架构改进报告：动态属性系统

基于动态属性测试结果的深度架构重构建议

1. 收拢状态管理 (Deepen CharacterStateManager)

涉及文件: backend/services/character_state.py, backend/routers/chat.py, backend/services/slash_handler.py
Strong (强烈推荐)

Problem (存在的问题)

CharacterStateManager 被设计为动态属性的状态机接缝，但它目前是一个Shallow Module（浅模块）。调用方（如 chat.py 中的 _build_full_system_prompt 和 slash_handler.py 中的 _stream_normal_chat_with_injection、_handle_reward）绕过了这个接口，直接使用裸 SQL (cursor.execute("SELECT value, social_status...")) 读写数据库。这导致了严重的Locality（局部性）缺失：如果未来增加新属性或修改数据库表结构，需要在多个文件中地毯式搜索并修改 SQL 语句。

Solution (解决方案)

强制所有涉及角色状态的读取和修改必须通过 CharacterStateManager 的接口。移除 chat.py 和 slash_handler.py 中的裸 SQL 查询，替换为对 CharacterStateManager.get_state() 和 .update_state() 的调用。

Benefits (收益)

  • Locality (局部性)：所有关于角色属性的规则、边界限制（如 0-100 的范围）和持久化逻辑全部收拢在一个文件中。
  • Leverage (杠杆率)：调用方只需获取字典对象，无需关心数据库连接的创建和销毁。
  • Testability (可测试性)：在编写测试时，可以通过 Mock CharacterStateManager 轻松注入状态，无需在每个测试用例中初始化 SQLite 内存数据库。

2. 解耦副作用管线 (Extract Side-Effect Handlers)

涉及文件: backend/services/slash_handler.py, backend/services/response_pipeline.py
Worth exploring (值得探索)

Problem (存在的问题)

slash_handler.py 中的 _register_pipeline_handlers 函数包含了所有 XML 标签（如 system_pass, glossary, new_course, explainer, property_update）的具体业务逻辑。这使得负责路由和指令分发的模块与具体的业务副作用紧密耦合。随着特性的增加，这个函数会无限膨胀。

Solution (解决方案)

将这些具体的 Handler 提取到一个独立的深模块中（例如 backend/services/pipeline_actions.py），提供统一的 register_all_handlers(pipeline) 接口。

Benefits (收益)

  • Seam (接缝)：将路由解析与副作用执行通过 Pipeline 注册机制彻底分离，形成清晰的接缝。
  • Testability (可测试性)：每个副作用处理器（如属性更新、生成玉简）可以被独立测试，而无需启动整个对话路由流。

👑 Top Recommendation (首选建议)

建议优先实施 1. 收拢状态管理 (Deepen CharacterStateManager)。

在当前的测试脚本 simulate_chat.py 中，为了重置环境，我们必须手动执行裸 SQL 更新数据库。在 chat.py 构建 System Prompt 时，也再次书写了相同的 SQL 语句去查询 value, social_status, social_skills, refractory_period。这违反了 DRY 原则，并且让 CharacterStateManager 形同虚设。

实施此重构不仅能大幅提升代码的局部性，还能消除潜在的数据库锁竞争/连接未关闭等隐患，为后续进一步丰富动态属性（如疲劳值、黑化值等）奠定极其坚实的基础。
