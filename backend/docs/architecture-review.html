代码架构深化分析报告

基于近期的 Bug 反馈与系统扩展需求，对现有架构的深度审视。

1. 聊天管道（ResponsePipeline）异常隔离与生命周期管理

涉及模块: backend/services/response_pipeline.py, backend/routers/chat.py

Strong (强烈推荐)

问题 (Problem)

当前的 ResponsePipeline 和 stream_chat 存在浅层封装问题。当生成器在迭代过程中遇到 GeneratorExit（前端断开连接）或内部工具抛出异常（如 process_memory_decay 耗时过长导致的前端超时断开）时，异常会直接向上传递，破坏整个 FastAPI 路由的协程状态，并引发僵尸进程或崩溃（正如 debug_error.log 中的 asyncio.exceptions.CancelledError）。

解决方案 (Solution)

将 ResponsePipeline 深化为一个具备完整生命周期管理的 Session Context Manager（会话上下文管理器）。将底层的流式迭代、异常捕获、资源释放（锁的释放、数据库连接关闭）封装在一个接缝 (seam) 之后。对外仅暴露一个安全的异步生成器接口。

收益 (Benefits)

高局部性 (Locality)：所有的流断开、网络波动异常处理集中在管理器内部，路由层无需关心底层的 GeneratorExit。高杠杆率 (Leverage)：未来添加任何新的流式工具（如长耗时的后台任务），都不会因为前端的意外断开而导致系统崩溃。

Before (浅层)

After (深化)

2. 动态属性的状态机接缝 (State Machine Seam)

涉及模块: backend/routers/course.py, backend/services/prompts.py, backend/database.py

Worth exploring (值得探索)

问题 (Problem)

随着 social_status, social_skills, 和 refractory_period 的加入，状态的更新逻辑散落在 course.py（供前端调用）和未来的后台任务中。目前这些逻辑是浅层的直接数据库 UPDATE。如果不加以约束，很容易出现数值越界、状态不一致（如高潮后未正确重置不应期），导致系统提示词注入混乱。

解决方案 (Solution)

引入一个专门的 CharacterStateManager 模块作为接缝。它对外暴露如 trigger_climax(), advance_round(), update_social_skills(delta) 等业务语义接口，而在内部维护数值边界和状态流转（例如 advance_round() 内部自动递减 refractory_period）。

收益 (Benefits)

局部性：所有关于“属性如何随时间/事件衰减和增长”的业务规则都集中在一处，修改不应期衰减公式只需改动一行代码。测试表面：我们只需测试 CharacterStateManager 的接口，即可验证复杂的动态属性变化逻辑。

Top Recommendation (首选推荐)

优先解决 1. 聊天管道异常隔离。从 debug_error.log 中明确看到 GeneratorExit 导致了底层的 CancelledError 并引发系统级崩溃。这是导致之前 /summarize 报错后整个服务卡死（FastAPI 崩溃）的根本原因。如果不解决这个接缝问题，任何耗时的后台处理（如复杂的 RAG 或记忆压缩）只要遇到前端刷新页面或断开连接，就会导致整个服务宕机。
